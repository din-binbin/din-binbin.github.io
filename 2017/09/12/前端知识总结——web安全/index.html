前端知识总结——web安全

发表于 2017-09-12

随着前端业务的发展，web的安全越来越重要，作为一名前端开发人员，必须知道相关的安全攻防知识。

使用cookie保存用户信息

HTTP是无状态的协议，因此需要使用一些方法去保存请求方的信息，cookie就是其中一种。

当第一次访问的时候，服务器会将带有cookie的返回结果返回给用户。
当第二次访问的时候，浏览器会将用户的cookie连同请求一起发送到服务器。

需要注意的是，cookie是在服务器端设置的，一般是通过登录请求的Ajax获取服务器的cookie。

XSS

伴随着web业务的迅速发展，涉及到的用户信息越来越多，为了保护用户隐私，web安全变得越来越重要。

常见攻击方式

  • sql注入（数据库查询语言注入，通过服务端防范）
  • XSS
  • CSRF
  • DDOS
  • HTTP劫持

XSS简介

XSS，即Cross-Site Script，跨站脚本攻击。

  • 攻击者向网页注入恶意脚本代码
  • 用户访问包含该恶意代码的网页，会获取到恶意脚本代码
  • 通过运行恶意脚本，获取和控制用户信息

通常有以下几种：

  • 反射型（非持久型）XSS
  • 存储型（持久型）XSS
  • DOM-Based型XSS

反射型和存储型均需要服务器的参与。

反射型（非持久型）XSS

诱导用户点击恶意链接来造成一次性攻击

  • 用户访问带有恶意脚本代码参数的URL
  • 服务器获取请求参数并且直接使用
  • 服务端”反射”回结果页
  • 用户被攻击

特点：一次性，通过链接点击引起。

存储型（持久型）XSS

恶意代码被存储到漏洞服务器中，用户浏览相关页面发起攻击

  • 将恶意代码上传或存储到漏洞服务器
  • 用户访问包含恶意脚本的页面
  • 服务器读取恶意数据并返回给用户
  • 用户被攻击

DOM-Based型XSS

基于浏览器DOM解析的攻击

  • 用户打开带有恶意脚本的链接
  • 浏览器在DOM解析时直接使用恶意数据
  • 用户被攻击

常触发场景：innerHTML、outHTML、document.write

XSS payload

实现XSS攻击的恶意脚本叫XSS payload。

XSS payload可实现：

  • 窃取用户cookie
    • document.cookie
  • 识别用户浏览器
    • navigator.userAgent
  • 伪造请求
    • get/post
  • XSS钓鱼
    • XSS+钓鱼

XSS 防御

XSS 防御 httpOnly

从攻击原理去防御

  • 可通过给cookie设置httpOnly属性，阻止客户端脚本访问cookie。
    • 缺点：自己也不能用；非根本性解决XSS。

XSS 防御 输入检查

要对用户的输入抱有怀疑的态度，对一些不正确不合规范的输入，需要做一些过滤和筛选。

  • 判断输入格式（白名单）
  • 过滤危险字符（黑名单）
  • 转义特殊字符（黑名单）

但是一些工具可绕过前端的检查，因此需要服务端结合前端来实现输入检查。

XSS 防御 输出检查

输出检查是最后一道防线，其根据不同场景对数据进行处理。

通常 XSS payload 会注入到以下几个环境：

  • HTML标签（HTML执行环境）
  • HTML属性（HTML执行环境）
  • script标签（JavaScript解析环境）
  • 事件属性（JavaScript解析环境）
  • URL（URL解析环境）

输出检查，就是为了避免将用户输入当成代码运行。对不同环境可采取：

  • HTML执行环境 ： 使用 HTMLEncode() 转义特殊字符
  • JavaScript解析环境 ： 使用 JavaScriptEncode() 转义特殊字符
  • URL解析环境 ： 使用 URLEncode() 转义特殊字符

web相关编码和转义

提到 Web 前端安全，第一个想到的就是 XSS。而提到 XSS，不得不说的就是编码了。

常用编码

  • URL 编码
  • HTML 编码
  • JS 编码

URL 编码

一般来说，URL只能使用英文字母（a-zA-Z）、数字（0-9）、-_.~4个特殊字符以及所有（;,/?:@&=+$#）保留字符。如果使用了一些其他文字和特殊字符，则需要通过编码的方式来进行表示。

在 URL 中传参是通过键值对形式进行的，格式上是以？、& 和 = 为特征标识进行解析，如果在键或者值的内容中包含一些特殊符号，就会造成解析错误。

对于上面的情况如果要正常解析，则需要进行编码，需要用不会造成歧义的符号代替有歧义的符号。

encodeURI

encodeURI 是用来编码 URI 的，最常见的就是编码一个 URL。encodeURI 会将需要编码的字符转换为 UTF-8 的格式。对于保留字符（;,/?:@&=+$#），以及非转义字符（字母数字以及 -_.!~*'()）不会进行转义。

在这里，%E5%B8%85 就是 帅 的 URL 编码，%E6%88%91 即为 我 的 URL 编码。然后由于 encodeURI 不转义&、?和=。所以对于 URL 参数的值是无法转义的。

encodeURIComponent

encodeURIComponent 是用来编码 URI 参数的。它会跳过非转义字符（字母数字以及-_.!~*'()）。但会转义 URL的 保留字符（;,/?:@&=+$#）。通常来说我们会 encodeURI 结合 encodeURIComponent 来使用。

需注意的是由于 encodeURIComponent 会编码所有的 URL 保留字，所以不适合编码 URL。

HTML编码

在 HTML 中，某些字符是预留的，比如不能使用小于号（<）和大于号（>），这是因为浏览器会误认为它们是标签。如果希望正确地显示预留字符，我们必须在 HTML 源代码中使用字符实体（character entities）。

当然还另一个重要原因，有些字符在 ASCII 字符集中没有定义，因此需要使用字符实体来表示，比如中文。

HTML 编码分为：

  • HTML 十六进制编码 &#xH;
  • HTML 十进制编码 &#D;
  • HTML 实体编码 &lt;

在 HTML 进制编码中其中的数字则是对应字符的 unicode 字符编码。
比如单引号的 unicode 字符编码是27，则单引号可以被编码为&#x27;

JavaScript转义

JavaScript 中有些字符有特殊用途，如果字符串中想使用这些字符原来的含义，需要使用反斜杠对这些特殊符号进行转义。我们称之为 Javascript编码。一般有以下几类：

  • 三个八进制数字，如果不够个数，前面补0，例如 “e” 编码为“\145”
  • 两个十六进制数字，如果不够个数，前面补0，例如 “e” 编码为“\x65”
  • 四个十六进制数字，如果不够个数，前面补0，例如 “e” 编码为“\u0065”
  • 对于一些控制字符，使用特殊的C类型的转义风格（例如\n和\r）

CSRF

CSRF，即Cross-Rite Request Forgery，跨站请求伪造。

CSRF 基本原理

  • 用户登录
  • 服务器返回cookie
  • 诱导访问攻击页面
  • 通过cookie伪造请求
  • 返回正常页面，发送伪造请求，用户被攻击

CSRF 防御

正常请求与伪造请求的不同：

  • 用户知情与否
    • 在发送请求时输入验证码可解决，但影响用户体验
    • 验证码一般是防止DDOS攻击
  • 是否跨站请求
    • 验证请求信息的referer属性可解决，但可能被篡改
  • 参数是否伪造
    • 可通过验证token解决，目前防CSRF最常用方法

CSRF 与 XSS 的结合

结合之后，攻击力会变得很强，可直接运行脚本代码发起攻击。

XSS蠕虫

蠕虫的传染机理是利用恶意脚本进行复制和传播。

条件：

  • 用户交互的页面
  • 影响范围广

XSS蠕虫会重复的发起XSS和CSRF攻击。

DDOS

DDoS，即Distributed Denial of Service，分布式拒绝服务。

DDOS 核心思想

通过大量非正常的请求来耗尽服务器的资源，使得服务器不能正常运行。

DDOS 防范

DDOS 攻击仍是业界安全问题中一个难题，当攻击流量超过了服务的网络设备的极限或带宽的最大负荷时，网站服务将瘫痪导致无法正常运作。

通常来说，对于 DDOS 攻击我们可以通过下面的方式来进行防范。

  • 验证码
    • 验证码是我们在互联网十分常见的技术之一。不得不说验证码是能够有效地防止多次重复请求的行为。目前来说，随着验证码方式的丰富化，导致黑客们没有一个比较有效的方式来获取和识别验证码。但由于验证码会影响到用户体验，因此验证码只有在比较重要的入口才会使用，如转账和登陆的场景。
  • 限制请求频率（ratelimit）
    • 限制请求频率是我们最常见的针对 DDOS 攻击的防御措施。其原理为设置每个客户端的请求频率的限制。简单来说，就是通过一些标识（如 IP 与 Cookie）来定位一个客户端，如果该客户端的请求在一定时间内过于频繁。则对该客户端之后的请求都重定向到一个出错的页面或者封杀相关 IP。
    • 当然许多框架都会有相应地限制请求频率的工具包，如 koa 框架便可以使用下面这些：koa-limit、koa-ratelimit。
  • 扩容加带宽
    • 简单来说就是，增加机器增加服务带宽。只要超过了攻击流量便可以避免服务瘫痪。在双十一活动或者12306 抢车票等场景时，网站服务都会对自己的机器进行扩容和提高带宽。然而为了避免由于使用过多机器导致成本太高且浪费的情况。通常来说都是根据网站活动和请求情况来实施相应扩容操作。
  • 其他的方法
    • 设置自己的业务为分布式服务，防止单点失效
    • 使用主流云系统和 CDN（云和 CDN 其自身有 DDOS 的防范作用）
    • 优化资源使用提高 web server 的负载能力

更多DDOS可参考：漫画告诉你什么是DDoS攻击、创业公司如何防御DDoS攻击

运营商HTTP劫持

在用户与服务器之间，存在着一个角色：运营商。

用户通过运营商发起请求，服务端通过运营商返回数据。

HTTP协议有这样一个特点，就是在传输过程中是明文的，不会加密。

有些运营商或者其他窃听者就会篡改网页内容，比如插入右下角小广告什么的，这就是HTTP劫持。

HTTPS

HTTPS可有效防止运营商HTTP劫持，防止页面篡改。

简单说，HTTPS = HTTP + SSL（安全套接层）。

SSL就是为了解决HTTP协议在传输过程中可能被篡改的问题而诞生的，后标准化为TSL（传输层安全协议），一般SSL/TSL并列出现。

有了SSL/TSL，就可以在服务端对要返回的数据进行加密，使得传输中的篡改变得困难，再到用户的浏览器进行解密。

对称加密和非对称加密

  • 对称加密
    • 加密和解密用的同一条秘钥
  • 非对称加密
    • 秘钥成对出现，通常其中一条不公开，叫私钥，另一个公开的叫公钥。其中一条加密的内容只能用另外一条解开。

非对称秘钥加密过程：

  • 请求HTTPS连接
  • 服务端返回公钥，保存私钥
  • 客户端产生对称秘钥，并使用服务器返回的公钥加密
  • 客户端发送加密后的对称秘钥到服务端
  • 服务端使用私钥解密得到对称秘钥
  • 服务端返回对称秘钥加密的页面
  • 至此，客户端和服务端可通过此对称秘钥通信

为了防止服务端向客户端发送的公钥不被窃取和伪造，需要保证服务器公钥的合法性。做法如下：

  • 服务端不再是直接返回公钥，而是先向CA请求，申请证书，再返回包含公钥的证书到客户端
  • 客户端收到证书后，会向CA询问并验证证书，以确保服务器的合法性
  • CA，即数字证书认证机构，具有权威性

其他安全

  • 点击劫持
    • 点击劫持实际上就是在正常界面上覆盖一层透明的内容，引导用户在不知情的情况下进行了某些操作，这些操作可能是广告点击，或者是开启摄像头等操作，甚至可能会导致一些数据别窃取，也有可能结合 XSS 和 CSRF 进行更负责的攻击。
  • 服务端安全
    • 除了前端注意安全，后端也需要对安全进行防范，常见比如防止 SQL 注入，登录会话管理等。
